# Recording Encryption Design

**Date:** 2026-02-17
**Status:** Approved

## Goal

Encrypt recordings on disk so they are unplayable outside the app and unrecognizable to casual snooping. Hash filenames to eliminate human-readable metadata from the filesystem.

## Threat Model

Casual snooping — prevent someone browsing files from recognizing or playing recordings (shared computer, roommate). Not intended to resist reverse-engineering or forensic analysis.

## Encryption

- **Algorithm:** AES-256-GCM via Apple CryptoKit
- **Key:** Single 256-bit `SymmetricKey`, generated on first launch, stored in Keychain via `KeychainHelper`
- **Per-file nonce:** Auto-generated by GCM. Stored as part of `SealedBox.combined` (nonce + ciphertext + tag)

## File Naming

- SHA-256 hash of `(original_filename + random_UUID)`, truncated to 32 hex chars
- Extension: `.glitcho`
- Example: `a7f3b21e9c4d8f0612ab3c5d7e9f1a2b.glitcho`

## Metadata Manifest

- `manifest.glitcho` in recordings directory
- AES-GCM encrypted JSON mapping hash filenames to metadata:
  ```json
  {
    "a7f3b21e...": {
      "channelName": "streamer",
      "date": "2026-02-17T14:30:45Z",
      "duration": 3600,
      "quality": "best",
      "originalFilename": "streamer_2026-02-17_14-30-45.mp4"
    }
  }
  ```
- Loaded into memory on app launch, written back after each recording completes

## Recording Pipeline

```
Streamlink → raw .mp4 (TS) → FFmpeg remux → encrypt → .glitcho → update manifest
```

Encryption hooks in after the existing remux step in `RecordingManager`. New method `encryptAndStoreRecording(at: URL) -> URL`:

1. Read remuxed MP4 data
2. Encrypt with AES-GCM
3. Write to `{hash}.glitcho`
4. Delete plaintext MP4
5. Add entry to manifest
6. Return encrypted URL

## Playback

```
Select recording → lookup manifest → decrypt .glitcho → temp .mp4 → AVPlayer → delete temp on close
```

- Temp files in `NSTemporaryDirectory()` with random names
- Cleaned up on player dismiss and on app launch (sweep stale temps)

## Export

- "Export" button in `RecordingsLibraryView`
- Decrypts to user-chosen location via `NSSavePanel` as standard `.mp4`

## Migration

- On app start, if `manifest.glitcho` does not exist, scan recordings directory for `.mp4` files
- Encrypt each, generate hash filename, add to manifest, delete original
- Non-blocking progress indicator ("Securing recordings...")
- Idempotent — interrupted migration resumes on next launch (remaining `.mp4` files get picked up)
- Skip active recordings (live Streamlink process); encrypt after they finish

## Edge Cases

| Scenario | Handling |
|----------|----------|
| Recording in progress during migration | Skip; encrypt after completion |
| Manifest corruption/deletion | Orphaned files; show "X unindexed recordings found" |
| Keychain key lost | Recordings unrecoverable (acceptable for threat model) |
| Disk full during encryption | Write fails, original `.mp4` preserved |
| Retention policies | Operate on manifest entries; delete both `.glitcho` and entry |
| Temp file cleanup | Sweep `NSTemporaryDirectory()` on launch for stale temps |
